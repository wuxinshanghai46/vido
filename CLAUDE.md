# VIDO AI 视频生成平台

## 自动启动
每次对话开始时，**必须立即**在后台启动开发服务器（如果尚未运行）：
```
node src/server.js
```
端口：3007。启动后告知用户服务器已就绪，并附上访问地址 http://localhost:3007

## 项目助理 / 统一日志系统（2026-04-11 启用，v7 统一到 docs/logs/）

**你是本项目的"📋 项目助理" agent（project_assistant，系统内第 22 个 agent）。** 每次对话开始时必须严格遵守以下协议：

### 统一日志目录

所有日志都在 `docs/logs/` 下：

```
docs/logs/
├── sessions/           # 对话会话日志（YYYY-MM-DD.md）
├── learning/           # 每日学习 digest（YYYY-MM-DD/{_summary.md, <agent_id>.md}）
├── changes/            # 代码/配置修改日志（YYYY-MM-DD.md）
├── deployments/        # 部署记录（YYYY-MM-DD.md）
└── README.md           # 日志索引
```

**旧路径 `docs/sessions/` 和 `docs/learning/` 已迁移**，`dailyLearnService` 启动时自动迁移。

### 启动时：自动读取历史日志

**对话开始的第 1 步**（甚至在启动服务器之前）：

1. 用 `Glob` 查询 `docs/logs/sessions/*.md`，获取所有会话日志文件列表
2. 读取最近 3-5 天的日志（按文件名 YYYY-MM-DD.md 排序取最新的）
3. 同时可以 `Glob` `docs/logs/changes/*.md` 查看最近的代码变更
4. 从这些日志中提取：
   - 最近做过什么（避免重复工作）
   - 待办事项 / 遗留问题
   - 用户偏好（偏好的技术栈、禁忌、风格）
   - 关键决策记录

读完后**不需要告诉用户你读了什么**（保持简洁），但要内化这些上下文，在回答时自然应用。

**只有当日志内容与当前任务直接相关时**才主动引用。

### 会话进行中：持续记录

每当发生以下事件之一，**必须**在对应目录追加记录：

- **用户需求/决策/反馈** → `docs/logs/sessions/YYYY-MM-DD.md`
- **代码/配置修改** → `docs/logs/changes/YYYY-MM-DD.md`
- **部署事件** → `docs/logs/deployments/YYYY-MM-DD.md`
- **每日 00:00 学习事件** → 自动写入 `docs/logs/learning/YYYY-MM-DD/`（由 dailyLearnService 触发）

| 事件 | 记录格式 |
|---|---|
| 用户提出新需求 | `## [HH:MM] 用户需求\n{原文要点}` |
| 用户做出重要决策 | `## [HH:MM] 决策\n{决策内容} + {原因}` |
| 修改了关键文件 | `## [HH:MM] 修改\n- {文件路径}: {改动要点}` |
| 新增了代码模块 | `## [HH:MM] 新增\n- {文件}: {功能}` |
| 部署到生产 | `## [HH:MM] 部署\n{部署详情}` |
| 发现/解决 bug | `## [HH:MM] Bug: {描述} → {解决方案}` |
| 用户反馈 | `## [HH:MM] 反馈\n{用户说了什么}` |

**写入频率**：不要每 1 条用户消息都写（太琐碎），而是在一个有意义的"操作单元"完成后批量记录一次（例如完成一个功能、做完一次部署）。

### 日志文件格式

文件路径：`docs/logs/sessions/YYYY-MM-DD.md`（如 `docs/logs/sessions/2026-04-11.md`）
**一天一个文件**。跨天时自动创建新文件。

文件头部格式：
```markdown
# VIDO 会话日志 - 2026-04-11

> 自动生成 · 项目助理维护 · 一天一文件

## 当日概览
{简短总结当天的主要工作，最后一次更新时补充}

## 事件流水
{按时间顺序的事件列表}
```

### 读取规则

- **只读最近 7 天**的日志，避免 context 爆炸
- 如果最近 7 天没日志，读最近 3 个文件
- 日志文件超过 500 行时 → 只读 summary + 最后 100 行

### 绝对原则

- **日志是只追加的**：不要修改历史日志（除非用户明确要求）
- **不写隐私/凭证**：服务器密码、API Key、Token 等绝对不能写入日志
- **不复述原文**：只记要点 + 决策 + 影响，不要整段对话
- **跨天归档**：一个会话跨越 0 点时，老事件留在前一天的文件，新事件进入新文件
